  • About this user guide

    • This document is an introduction to the CEEQ - Comprehensive End-to-End Quality evaluation tool. There are some previous web application experience are presumed, such as: web search, web explore etc.

    This user guide does not cover 100% of the features, however, it will explain all the main concepts and help you get started. At the meantime, it provides other learning resources for your further reading. This material will help you be ready to begin exploiting our tools.

  • What is CEEQ?

    • Comprehensive End-to-End Quality is the novel framework to describe the measurement of West developed solutions based on tracked issues in JIRA system.

    CEEQ ranges from 0 -10 to illustrate the quality of West developed solutions. CEEQ helps our testing procedure on assembled West developed solutions become more quick, easy and accurate through software lifecycle.

    As somebody says: "You cannot control what you cannot measure." Measurement is the process by which numbers or symbols are assigned to attributes of entities in the real world in such a way as to describe them according to clearly defined unambiguous rules

    The Risk management of our testing procedure indicates that West developed solutions failure has caused more than inconvenience. The causes have ranged from poorly designed user interface to direct programming errors.

    Also, an West developed solutions with good structural quality costs less to maintain and is easier to understand and change in response to pressing business needs.

    The CEEQ is expected to be useful in evaluating our internal West developed solutions from begin to end.

JIRA Issues Weight Factor  

Our framework takes the data from JIRA to evaluate the CEEQ score of West developed solutions. There five priorities are included in CEEQ, the ratio of these priorities is 5:4:3:2:1. The following different priority of the issue are taken account into our framework:

  • Blocker:

    • Highest priority. Indicates that this issue takes precedence over all others.

  • Critical:

    • Indicates that this issue is causing a problem and requires urgent attention.

  • Major:

    • Indicates that this issue has a significant impact.

  • Minor:

    • Indicates that this issue has a relatively minor impact.

  • Trivial:

    • Lowest priority.

CEEQ Framework Scope  

It won't be nesseary to track all issues in JIRA into our framework.

  • Issue Types:
    • Types included
      • Bug
    • Types not included
      • New Feature
      • Task
      • Improvement
      • Suggested Improvement
      • Environment
  • Issue Status:
    • Open
      • Open
      • In Progress
      • Reopen
      • Discovery
      • Review
      • Pending
      • Research
    • Resolved
      • Resolved
      • UAT Testing
      • Done
    • Closed
      • Closed
  • Issue Resolution:
    • Resolution not included if it is closed
      • Duplicate
      • Works as Design
  • Custom Field Excluded:
    • Resolution: External Limitation
    • TFCC IS Root Cause: Test Data
    • Issue is closed
  • Workflow Metatype
    • We also provide different scope of JIRAs based on Workflow Metatype property as follow:
    • Include UAT: JIRAs with Workflow Metatype as UAT are counted
    • Exclude UAT: JIRAs with Workflow Metatype as UAT are not counted
    • Only UAT: JIRAs with Workflow Metatype as UAT are only counted
  • Multiple Components
    • If a ticket has more than one components, we take first component which is under our framework:
    • If the first component is under our framework, then take the first component
    • If the first component is not under our framework, then check next one
      • If the second component is under our framework, then take the second component
      • If the second component is not under our framework, then check next one
      • Until check the last component
    • If none component is under our framework, then no component is selected for this ticket
  • Custom Date Range
    • We provide specific user the ability to select their own date range to track JIRA tickets.
    • The default date range is last 30 days up to today.
    • System gives user several shortcuts of date range:
      • Today
      • Yesterday
      • Last 7 Days
      • Last 30 Days
      • This Month
      • Last Month
    • User could select any date range from Custom Range
    • For custom date range, we track Internal Testing and UAT tickets only.

Voice Slots Percentage Range - New approach  

Voice Slots has three sub components: English, Spanish, Other, each sub component only has 1 ‘Missing’ and 1 ‘Defective’ issues. Issue is Open when percentage of ‘Missing’ or ‘Defective’ reaches following range. Issue will be changed from ‘Open’ to ‘Resolved’ as ‘Trivial’ until all verbiages are cleared.

Software Testing Characteristics  

Software Testing Characteristics considers the application of those techniques in evaluating the principal attributes used to describe the quality of software applications or systems.

  • Quality attributes for domain testing

    • Functional testing is focused on "what" the product does

    The following quality attributes are considered:

    • Accuracy

      • considers the application of those techniques in evaluating the principal attributes used to describe the quality of software applications or systems.

    • Suitability

      • Suitability testing involves evaluating and validating the appropriateness of a set of functions for its intended specified tasks.

    • Interoperability

      • Interoperability testing tests whether a given application can function correctly in all intended target environments (hardware, software, middleware, operating system, etc.).

    • Functional cecurity

      • Functional security testing (penetration testing) focuses on the ability of the software to prevent unauthorized access, whether accidental or deliberate, to functions and data.

    • Usability

      • Usability testing measures the suitability of the software for its users, and is directed at measuring the following factors with which specified users can achieve specified goals in particular environments or contexts.

    • Accessibility

      • It is important to consider the accessibility of software to those with particular requirements or restrictions in its use. This includes those with disabilities.

  • Quality attributes for technical testing

    • Quality attributes for Technical Test Analysts focus on "how" the product works, rather than the functional aspects of "what" it does. These tests can take place at any test level, but have particular relevance for Component test

    The following quality attributes are considered:

    • Technical Security

      • The principal concern regarding security is to prevent information from unintended use by unauthorized persons. Security testing attempts to compromise a system’s security policy by assessing a system’s vulnerability to threats.

    • Reliability

      • An objective of reliability testing is to monitor a statistical measure of software maturity over time and compare this to a desired reliability goal.

    • Efficiency

      • The efficiency quality attribute is evaluated by conducting tests focused on time and resource behavior. Efficiency testing relating to time behavior is covered below under the aspects of performance, load, stress and scalability testing.

    • Maintainability

      • Maintainability tests in general relate to the ease with which software can be analyzed, changed and tested. Appropriate techniques for maintainability testing include static analysis and checklists.

    • Portability

      • Portability tests in general relate to the ease with which software can be transferred into its intended environment, either initially or from an existing environment. Portability tests include tests for installability, co-existence/compatibility, adaptability and replaceability.
